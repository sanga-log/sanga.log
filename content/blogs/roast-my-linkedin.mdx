## 시작은 딴 나라 얘기였습니다

미국 개발자 커뮤니티에서 요즘 "Roast My ~" 시리즈가 한창 화제입니다. Roast My Strava, Roast My Code, Roast My Github... AI한테 내 결과물을 가져다 주면 신랄하게 까달라고 하는 콘셉트입니다.

보면서 딱 생각했습니다. **이거 재밌겠다!**

마침 그 날 LinkedIn 프로필을 수정했어서인지 바로 눈에 들어왔습니다. LinkedIn 프로필 팩폭 서비스! 우리나라에도 있을까 싶어서 찾아봤는데 아직은 없었고, 해외에 있는 비슷한 서비스들도 직접 들어가봤더니 하나같이 작동을 안 했습니다. 스크래핑 API 할당량이 다 찼거나, LinkedIn이 봇 접근을 아예 막아버린 거였습니다.

그래서 바~로 만들기로 맘먹었습니다! 이왕이면 한국식으로! 🇰🇷

## 첫 번째 기술 결정 — 스크래핑을 포기하지 않는다

LinkedIn은 스크래핑을 철저하게 막습니다. 실제로 소송까지 간 케이스가 있을 정도입니다. 근데 찾아보니 아직 가능한 API들이 있었습니다.

그래서 Claude와 기본 UI부터 빠르게 만들어서 스크래핑이 어떻게 진행되는지 어떤 에러가 나는지 테스트를 해보았는데, 하다보니 고민이 생겼습니다.

API가 막히거나 할당량이 다 차면 어떡하지?

그래서 RapidAPI 마켓플레이스를 뒤지기 시작했습니다. LinkedIn 스크래핑 API를 **하나하나 다 찾아봤습니다.** 유료는 사용자가 급속도로 늘어났을 때 고려하기로 하고, 우선 무료를 열심히 찾아서 8개를 모았습니다! 하나의 RapidAPI 키로 여러 API를 무료 구독할 수 있다는 걸 알게 되면서, 하나가 한도에 걸리면 다음 API로 넘기는 fallback chain 구조를 만들었습니다. 이렇게 할당량 문제는 해결!🌞

그리고 스크래핑 차단 문제는 더 재밌게 풀어보았습니다. **막히는 경험 자체를 재미있는 UX로 만들어버리자!** 에러 페이지를 보여주는 대신, 차단당하는 순간을 할미 말투로 풀어내면 오히려 더 웃기지 않을까?

```
└─ 8개 RapidAPI를 순차적으로 시도
    ├─ 성공 → AI 분석으로 이동
    └─ 전부 실패
         ├─ 429 할당량 초과 -> "할미떼가 너무 인기가 많아부러서 요청 한도를 넘겨부렀어유~"
         ├─ 403 스크래핑 차단 -> "LinkedIn이 할미 못 들어오게 막아부렀어유~"
         └─ PDF/스크린샷 업로드로 안내
```

사용자 입장에서는 이 과정이 전혀 안 보입니다. 내부에서 조용히 돌다가, 정말 안 될 때만 할미 말투로 안내하고, 거기엔 항상 두 번째 방법(직접 업로드)이 기다리고 있습니다. 차단당하는 게 실패가 아니라 **스토리가 되는 구조**입니다. 이 아이디어가 떠올랐을 때 얼른 사용자들에게 보여줄 생각에 신나서 작업했던 것 같습니다.

## 두 번째 기술 결정 — PC와 모바일을 다르게

거의 다 만들어갈때쯤 테스트하다보니 하나가 걸렸습니다. LinkedIn 프로필을 PDF로 저장하는 기능이 **PC에서만 됩니다.** 모바일 앱에는 그런 기능이 없습니다.

잠깐 고민했지만, 그러면 둘 다 지원하면 되지! 하는 마음으로 바로 작업에 들어갔습니다.

- **💻 PC**: LinkedIn → 더보기 → PDF로 저장 → 업로드
- **📱 모바일**: 프로필 스크린샷 촬영 → 업로드 → AI Vision으로 분석

PDF는 텍스트가 그대로 뽑히니까 분석 정확도가 높고, 스크린샷은 AI의 이미지 분석 능력을 쓰는 겁니다. 어떤 방법이든 같은 `/api/roast` 엔드포인트로 들어오고, 파일 타입에 따라 처리만 달라지게 만들었습니다.

## 아키텍처 설계 — 토스 Frontend Fundamentals 적용

폴더 구조를 잡을 때 [토스 Frontend Fundamentals](https://frontend-fundamentals.com)를 참고했습니다. 핵심은 간단합니다 — **함께 수정되는 코드는 같은 디렉토리에.** 예를 들어 결과 화면을 고치면 `ResultScreen.tsx`와 `CringeGauge.tsx`는 거의 항상 같이 수정되는데, 이 둘이 다른 폴더에 있으면 왔다갔다 하느라 정신없습니다.

```
app/
├── components/           # 전역 공유 UI (2개 이상 도메인에서 사용)
│   └── HalmiSpinner.tsx
│
├── domains/              # 화면별 도메인 (함께 수정되는 코드 응집)
│   ├── result/
│   │   ├── ResultScreen.tsx
│   │   └── CringeGauge.tsx  # result 전용 — 다른 곳에서 안 씀
│   └── upload/
│       ├── UploadScreen.tsx
│       └── UploadZone.tsx
│
├── lib/                  # 전역 공유 타입/유틸
└── api/                  # API 라우트
```

판단 기준은 딱 하나였습니다. "이 컴포넌트 다른 화면에서도 쓰나?"

쓰면 `components/` 안 쓰면 해당 `도메인 폴더 안`. 심플하니까 좋습니다.

## 런칭 후 — 실서비스에서 터진 것들

배포하고 하루 만에 코드로는 절대 잡을 수 없었던 문제들이 발생했습니다. 분명 이상없었는데...

### 1. 한글 LinkedIn URL 스크래핑 실패

`상아-이-1234567` 같이 한글이 섞인 username을 넣으면 8개 API가 전부 실패합니다. RapidAPI 스크래퍼들이 비영문 username을 아예 못 처리하는 거였습니다.

에러 화면보다 사전 안내가 낫다고 봤습니다. 한글이 감지되면 "스크래핑이 안 될 수 있어유" 문구를 미리 띄워서, 업로드로 자연스럽게 유도하도록 바꿨습니다.

### 2. Gemini JSON 파싱 실패

간헐적으로 500이 터졌습니다. Gemini가 JSON을 반환할 때 가끔 `json ... ` 마크다운 코드블록으로 감싸거나, 앞뒤에 설명 텍스트를 붙이는 경우가 있었습니다. 단순 `JSON.parse()`로는 당연히 터집니다.

2단계 파싱으로 해결했습니다:

````typescript
try {
  // 1차: 코드블록 제거 후 파싱
  const cleaned = text.replace(/```[\s\S]*?\n([\s\S]*?)```/g, "$1").trim();
  result = JSON.parse(cleaned);
} catch {
  // 2차: 응답에서 첫 번째 { ~ 마지막 } 를 직접 추출
  const firstBrace = text.indexOf("{");
  const lastBrace = text.lastIndexOf("}");
  if (firstBrace !== -1 && lastBrace > firstBrace) {
    result = JSON.parse(text.slice(firstBrace, lastBrace + 1));
  }
}
````

1차에서 정규식으로 코드블록을 제거하고, 실패하면 2차에서 JSON 객체의 시작(`{`)과 끝(`}`)을 직접 찾아 추출합니다. Gemini가 코드블록 앞뒤에 어떤 텍스트를 붙이든 JSON만 정확히 꺼낼 수 있습니다. 파싱 실패 시에는 raw text를 Sentry에 찍어서 나중에 뭐가 깨뜨렸는지 추적할 수 있게 했습니다.

### 3. Gemini가 날짜를 모릅니다!

사용자 피드백으로 알게 된 문제입니다. 프로필에 "삼성 청년 SW 아카데미 2025년 1월"이라고 적혀 있으면, Gemini가 이걸 미래 일정으로 오인해서 "시작도 안 한 걸 경력이라고 올려 놓으면 사람들이 알아주는 줄 아는겨?"라고 팩폭하는 상황이 발생했습니다. 2025년 1월은 이미 지난 과거인데도.

LLM은 학습 시점의 지식만 가지고 있어서, 프롬프트에 현재 날짜를 명시적으로 알려주지 않으면 시간 기준을 잡지 못합니다. 해결은 간단합니다.

```typescript
function buildSystemPrompt() {
  const today = new Date().toISOString().split("T")[0];
  return `...
오늘 날짜: ${today}
프로필에 적힌 날짜/기간을 판단할 때 반드시 오늘 날짜 기준으로 과거/현재/미래를 판단하세요.
...`;
}
```

한 줄 추가로 해결됐지만, 직접 피드백을 받기 전까지는 전혀 떠올리지 못한 문제였습니다. 실제로 사용자의 피드백을 통해 문제점들을 파악할 수 있어서 피드백 영역을 추가하길 정말 잘했단 생각이 들었습니다.

### 4. 클립보드 권한 거부

결과 화면 공유 버튼에서 `NotAllowedError`가 터졌습니다. `navigator.share()` → 실패 → `clipboard.writeText()`를 순서대로 호출하는 구조였는데, 버튼 클릭 한 번으로 보안 API를 두 개 연달아 부르면 두 번째에서 제스처가 소비된 걸로 간주돼서 거부당합니다.

`navigator.share` 지원 환경이면 그것만 쓰고 바로 `return`, 미지원이면 클립보드로 분기하게 고쳤습니다. 두 API가 경쟁하는 구조를 없앤 겁니다.
